Phase 01: Extract

非結構化文本提取

使用 PyMuPDF 強力解析原始 PDF 檔案，將二進制文件轉化為可處理的純文本字串。此階段負責掃描整個資料夾並建立初步的文件字典。

Output Data Structure
[
  {
    "filename": "manual.pdf",
    "content": "長文本內容..."
  }
]
Phase 02: Transform

文本切片與元數據豐富

透過 RecursiveCharacterTextSplitter 進行智能切片。這不是簡單的字數切割，而是會尋找換行與句號，確保語義完整性，同時自動注入原始路徑標籤。

Processed Chunk Structure
{
  "content": "500字精華片段",
  "metadata": {
    "source": "manual.pdf",
    "path": "Data/manual.pdf"
  }
}
Phase 03: Load

向量化與持久化載入

呼叫 Embedding Model (多國語言模型) 將文字映射到高維空間座標，並存入 ChromaDB。這讓系統具備了「根據意思」搜尋資料的能力。

Database Storage Map
ids: ["id_0", "id_1"...],
documents: ["文本片段..."],
metadatas: [{source...}],
embeddings: [[0.1, -0.4...]]

數據架構映射 (Data Architecture Mapping)

檔案 方法或函數 功能 資料 資料結構
batch_read_pdf.py batch_process_manuals Extract (提取)：讀取所有 PDF的長文 all_data 字典 :
"content": PDF文字
"filename": PDF檔案名稱
"path": PDF檔案完整路徑
batch_chunk_text.py create_chunks_with_metadata Transform (轉換)：將長文切塊 processed_chunks 字典 :
"content": all_data的content ---> 切成固定大小的文字檔案(長度500，重複50)
"metadata": {"source": all_data的filename, "path": all_data的path}
vector_storage.py save_to_vector_db Load (載入)：存入 ChromaDB 向量資料庫 chromadb>plc_manuals 資料表 :
ids: id_num一個不會重複的字串，作為主索引
documents: processed_chunks的content
metadatas: processed_chunks的metadata
embeddings: 針對內容計算出的向量數值
run_pipeline.py start_pipeline 將以上三個程序整合起來執行
⚙️

總控核心：run_pipeline.py

這是整個系統的指揮官。它不負責具體的資料處理，而是負責**調度與編排 (Orchestration)**。它確保資料嚴格依照 Extract ➔ Transform ➔ Load 的順序流動，並處理執行過程中的異常與進度回報。
