[遇到的問題]：
物理空間膨脹：執行 delete_collection 雖能在邏輯層（SQL 記錄）清空資料，但在 Windows 環境下，硬碟中存放向量索引的 UUID 資料夾不會立即消失，反而會隨執行次數增加而產生多個實體碎片資料夾。
檔案鎖定 (File Locking)：因 Python 行程或資料庫連線尚未完全釋放，導致作業系統限制了刪除實體檔案的權限。
非同步清理：ChromaDB 的設計傾向於效能優先，將實體檔案的清理/合併動作延後執行，而非即時與邏輯刪除同步。
[進階的解決方案]：
使用 Upsert 替代 Delete：將 collection.add 改為 collection.upsert。透過內容產生的唯一 Hash 值作為 ID，重複執行時會直接更新原索引片段，而非產生新的實體資料夾碎片。
顯式關閉 Client 連線：在程式結束前確保呼叫相關釋放機制，或在特定維護週期內重啟服務，以利系統執行垃圾回收（GC）與磁碟空間整理。
手動環境重置 (Reset)：在開發階段，若需 100% 乾淨的狀態，可直接透過 Python 指令 shutil.rmtree() 刪除整個 my_fatek_db 資料夾。
容器化部署：未來若部署於 Linux 或 Docker 環境，其檔案系統處理機制較為穩健，可減少 Windows 特有的檔案鎖定殘留問題。

[進階課題 1]：深入解析 AI API 回傳格式 (OpenAI 相容模式)
當你執行 response.json() 時，你會得到一個符合工業標準的 JSON 物件。了解它能讓你更精確地控制 AI 的行為與成本。
[建議學習方向]
完整 JSON 結構拆解：
id 與 created：學習如何利用唯一識別碼與 Unix 時間戳記進行日誌紀錄 (Logging)。
model：確認回傳結果確實來自你指定的模型，避免調用錯誤。
usage 數據監控：重點研究 prompt_tokens 與 completion_tokens。這對於評估「切塊長度」對運算成本（或地端硬體壓力）的影響至關重要。
finish_reason 判斷：了解 AI 停止回答的原因（是正常結束 stop，還是因為長度超出限制 length）。
多重選項 (N-Choices) 實驗：試著將請求參數 n 設為大於 1，觀察 choices 列表如何產生多個不同的 message 內容，並思考在 PLC 故障排除場景中，提供「多個建議方案」的實用性。

[進階課題 2]：ChromaDB 資料表結構與向量屬性意義
ChromaDB 不僅僅是一個存儲文字的地方，它是一個具備「空間座標」概念的數據庫。
[建議學習方向]
四大核心組件的關係：
ids：學習如何自定義具備唯一性的 ID（例如使用檔案 MD5 或 UUID），以實現高效的數據更新與刪除。
embeddings：研究浮點數向量（Vector）的本質。你可以試著將向量印出來看，理解這串數字是如何代表「語意座標」的。
metadatas (中繼資料)：這是 RAG 的靈魂。練習如何存入更多維度（如頁碼、建立日期、設備型號），並學習使用 where 條件進行「精確篩選 + 語意搜尋」的複合查詢。
documents：理解原始文本如何與向量建立一對一映射。
距離算法 (Distance Metrics)：
研究 l2 (歐式距離)、ip (內積) 與 cosine (餘弦相似度) 的差異。
實驗：改變資料庫初始化時的距離算法，觀察對於同一個 PLC 問題（如：高速計數器接線），分數（Score）的變動規律。
持久化機制：研究 PersistentClient 在硬碟中產生的 .sqlite3 與索引檔案，了解資料是如何在關機後依然被保存的。
